## Задача 2. Выбор организации серверов

### 1. Высоконагруженная база данных MySQL, критичная к отказу
* **Выбор**: **Физические сервера**.
* **Критерии**: Базы данных под высокой нагрузкой требовательны к скорости дисков и процессора. Физические серверы дают максимальную производительность без накладных расходов на виртуализацию. Отказоустойчивость при этом настраивается на уровне самой СУБД (например, репликация).

### 2. Различные web-приложения
* **Выбор**: **Виртуализация уровня ОС (контейнеризация)**.
* **Критерии**: Контейнеры легковесны, делят ядро хоста и быстро запускаются. Это позволяет эффективно использовать ресурсы сервера для множества независимых приложений и упрощает процесс деплоя.

### 3. Windows-системы для использования бухгалтерским отделом
* **Выбор**: **Паравиртуализация**.
* **Критерии**: Для работы Windows-систем и бухгалтерского ПО (например, 1С) нужна полноценная ОС Windows. Паравиртуализация (Hyper-V, KVM) позволяет изолировать рабочие места пользователей и легко делать резервные копии (снапшоты) перед обновлениями.

### 4. Системы, выполняющие высокопроизводительные расчёты на GPU
* **Выбор**: **Физические сервера**.
* **Критерии**: Расчетам на GPU нужен прямой доступ к видеокарте через шину PCIe. Виртуализация усложняет проброс видеокарт и может снижать производительность, поэтому для таких задач используют физическое железо.

## Задача 3. Выбор системы управления виртуализацией

### Сценарий 1 (100 ВМ, Windows-инфраструктура, балансировка, резервное копирование)
* **Решение**: **Microsoft Hyper-V** (в составе Windows Server) или **VMware vSphere**.
* **Обоснование**: Отличная интеграция Hyper-V с Windows Server и Active Directory. Встроенные средства (Hyper-V Replica, Failover Clustering) и сторонний софт (например, Veeam) полностью закрывают задачи резервного копирования и отказоустойчивости.

### Сценарий 2 (20-30 серверов, Linux/Windows, бесплатное open source решение)
* **Решение**: **Proxmox VE (на базе KVM)**.
* **Обоснование**: Полностью бесплатная платформа с открытым кодом. KVM дает высокую производительность. Proxmox предоставляет удобный веб-интерфейс для управления кластером, дисками и встроенным бэкапом из коробки.

### Сценарий 3 (Бесплатная и производительная виртуализация Windows)
* **Решение**: **Microsoft Hyper-V Server**.
* **Обоснование**: Родной гипервизор от Microsoft обеспечивает максимальную совместимость и производительность для Windows-систем. Автономная версия Hyper-V Server бесплатна.

### Сценарий 4 (Тестирование на нескольких дистрибутивах Linux)
* **Решение**: **LXD / LXC**.
* **Обоснование**: Системные контейнеры запускаются за секунды и используют ядро хоста, что экономит ресурсы. Это позволяет быстро развернуть чистые окружения разных дистрибутивов Linux на одной машине.

## Задача 4. Гетерогенные среды виртуализации

### 1. Проблемы гетерогенной среды
* **Сложность поддержки**: инженерам нужно знать особенности работы разных гипервизоров.
* **Трудности с миграцией**: перенос ВМ между разными платформами требует конвертации дисков и переустановки драйверов гостевой ОС.
* **Разрозненный мониторинг и бэкап**: сложнее настроить единые правила резервного копирования и отслеживания метрик.
* **Разделение ресурсов**: нельзя быстро перераспределить мощности между разными кластерами виртуализации.

### 2. Минимизация рисков
* Использование единых систем мониторинга (например, Prometheus + Grafana, Zabbix).
* Управление инфраструктурой через IaC (Terraform, Ansible).
* Применение универсальных систем бэкапа (например, Veeam).

### 3. Решение (создавал бы или нет)
Для небольших и средних компаний я бы выбирал гомогенную (однородную) среду, так как её проще и дешевле поддерживать. Гетерогенную среду стоит создавать только при наличии четких требований: например, для экономии на лицензиях (Windows-нагрузки на Hyper-V, Linux-нагрузки на KVM) или в процессе миграции со старой платформы на новую.
